## Repository Findings

1. `secrets` was not fully ignored by Git. The root `.gitignore` only ignored selected file extensions, so nested config trees like `secrets/PROXY/` and `secrets/ROUTE42/` still appeared in `git status`.
2. `secrets/README.md` and `secrets/print-github-secrets.sh` are already tracked by Git. Adding `/secrets/` to `.gitignore` protects new files, but tracked files stay tracked until they are removed from the index in a separate Git cleanup step.
3. The repo contains multiple local notes about the new VPS, but they do not agree on one authoritative client profile.

The repo currently contains three conflicting client variants:

3. `secrets/PROXY/xray/config.vultr-ru-direct.json`
   - points to `108.61.171.121`

   - adds broader direct-routing rules for RU traffic

There is also a fourth conflicting source:

- `secrets/client_config.md` documents `108.61.171.121` with yet another Reality tuple that does not match `config.hostkey.json` or `config.vultr-ru-direct.json`

## Root Cause

The new VPS itself is not the primary failure point.

The direct root cause of the browser failure was that the local client side had multiple conflicting configs for `108.61.171.121`, while the macOS LaunchAgent `local.xray` was configured to always start:

- `/Users/sergeibystrov/PROJECTS/test/VPNCLIENT/secrets/PROXY/xray/config.json`

So if a user edited or tested `config.hostkey.json` while `launchctl` still ran `config.json`, the browser would continue using the wrong profile.

Practical reproduction on 2026-03-22:

- live server tuple on `108.61.171.121` works when used in a temporary local client config
- `secrets/PROXY/xray/config.vultr-ru-direct.json` reproduces the hang on HTTPS requests
- `secrets/client_config.md` reproduces the same hang on HTTPS requests
- `secrets/PROXY/xray/config.vultr.json` reproduces a TLS reset because it still uses plain TLS instead of the live Reality tuple

## What This Means

The repo preserves useful diagnostic evidence for `108.61.171.121`, but the practical decision is now simple:

- keep the server configs and notes
- do not use the Vultr server for daily traffic
- do not promote it to production

Why the decision is firm now:

1. real browser use after switching to the new server was bad enough to force a manual rollback
2. a later canary run on `2026-03-22 11:07` timed out on HTTP and HTTPS before any live cutover was allowed
3. that canary failure means the route is still not safe even when isolated from the production config

Supporting evidence:

- SSH to `108.61.171.121` shows `xray.service` active and listening on `:443`
- the server can resolve DNS and reach `https://www.google.com` and `https://ifconfig.me` directly
- the server journal shows accepted proxy traffic from the client IP `91.218.116.71`
- the live server-side tuple matches `secrets/PROXY/xray/config.hostkey.json`
- the older `secrets/walkthrough.md` and `secrets/client_config.md` do not match the live server-side tuple
- the isolated local canary for `config.hostkey.json` still produced browser-path timeouts on `2026-03-22 11:07`

## Decision Guidance For The New VPS

Current recommendation: do not promote `108.61.171.121` to the default production server.

Reason:

- a real local switch of `local.xray` to the new VPS was attempted on 2026-03-22
- proxy traffic was observed in the local access log after the switch
- despite that, real browser use was judged bad enough that the setup was rolled back manually to `5.39.219.74`

So the new VPS can remain an archived lab candidate, but it is not a safe replacement for the baseline.

Provider-side status on `2026-03-22`:

- the only Vultr Cloud Compute instance for `108.61.171.121` was deleted through the Vultr API
- post-delete verification showed no remaining Vultr `instances`, `snapshots`, `blocks`, or `reserved_ips`
- local repo files for the Vultr attempt remain intentionally preserved as forensic and migration notes

Do not use these as authoritative client profiles for the new VPS:

- `secrets/PROXY/xray/config.vultr.json`
- `secrets/PROXY/xray/config.vultr-ru-direct.json`
- `secrets/client_config.md`

Use `secrets/PROXY/xray/config.hostkey.json` as the best local reference for the new VPS if future investigation resumes, but treat it as experimental and non-production.

Retire it if:

- you want the fastest path back to stability
